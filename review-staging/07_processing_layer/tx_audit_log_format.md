## Module: Transaction Audit Log Format
- **Layer**: Processing Layer — AST (Aros Studio Tokenomics)
- **Status**: Production-grade
- **Author**: Aros Studio NodeChain Division
- **Last Updated**: 2025-07-05

---

## Overview

The `tx_audit_log_format` module defines the canonical structure and encoding rules for audit records generated by the AST system. Unlike the journal (which tracks the execution path of each individual transaction), the audit log provides a **comprehensive, system-level record** of all security-relevant events, policy violations, guardrail triggers, validation anomalies, and rollback incidents.

These records are designed for compliance, forensic recovery, and retrospective analysis. All logs are signed, hash-linked, and stored immutably.

---

## Purpose

- Record critical events affecting TX lifecycle integrity
- Enable external audit and compliance review
- Maintain traceable lineage for rollback and dispute resolution
- Track security events, misbehavior, or node anomalies
- Support cross-node consistency verification

---

## Audit Log Schema

Each audit entry includes the following fields:

| Field               | Description                                                       |
|---------------------|-------------------------------------------------------------------|
| `log_id`           | Unique identifier for the audit event                              |
| `tx_id`            | Associated transaction (if applicable)                             |
| `event_type`       | One of: `validation_error`, `rollback`, `guardrail`, `policy_breach`, `system_fault` |
| `node_id`          | Node that generated the audit event                                |
| `timestamp`        | Unix timestamp of the event                                        |
| `severity`         | One of: `info`, `warning`, `critical`                              |
| `details`          | Human-readable explanation or system message                      |
| `snapshot_id`      | Snapshot reference (if event is TX-related)                        |
| `hash_chain_prev`  | Hash of previous audit entry (per node)                            |
| `hash_current`     | Current entry hash (to validate chain integrity)                   |
| `signature`        | Digital signature of the record (signed by node’s private key)     |

## Sample Entry

```json
{
  "log_id": "AL-45329",
  "tx_id": "TX-9183-AST",
  "event_type": "guardrail",

  "snapshot_id": "SS-2038",
  "hash_chain_prev": "0xa1b2c3...",
  "hash_current": "0xd4e5f6...",
  "signature": "0xBASE64..."
}

```

---

## Event Types

| Type | Description |
| --- | --- |
| `validation_error` | Any failure in validation pipeline that is flagged for escalation |
| `rollback` | Rollback event triggered after partial execution |
| `guardrail` | Triggered guardrail action (reject, delay, redirect) |
| `policy_breach` | Attempt to violate emission limits, jurisdictional bans, or protocol caps |
| `system_fault` | Internal module error, desync, memory pressure, snapshot corruption |

---

## Severity Levels

| Level | Use Case Example |
| --- | --- |
| `info` | Normal operation, e.g., simulation path taken |
| `warning` | Non-fatal anomaly, e.g., high risk score but passed guardrail |
| `critical` | System integrity issue, invalid snapshot, signature mismatch, rollback triggered |

---

## Log Storage and Protection

- Audit logs are stored in **append-only**, hash-chained segments
- Each node maintains its own signed chain of events
- Periodic Merkle roots can be exported to external anchor points for notarization
- Logs cannot be edited, redacted, or overwritten — violation causes node disqualification

---

## Cross-Node Synchronization

- Audit logs from multiple nodes can be compared using:
    - Time window correlation
    - Snapshot ID matching
    - Cross-validation of hash chains
- Used for fork detection, misbehavior analysis, or legal investigation

---

## Developer Notes

- All audit log emitters must sign the record before storing it
- Never reuse `log_id`; each must be globally unique
- Logging is non-blocking — failure to write a log cannot block TX processing, but must emit fallback signal
- A separate `audit_validator` module may be developed to continuously scan logs for anomalies
